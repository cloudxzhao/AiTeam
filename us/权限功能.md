- ### **产品需求文档 (PRD)：统一权限与用户管理系统**

  **文档版本：** 1.1  
  **发布日期：** 2026年3月9日  
  **项目名称：** 统一权限与用户管理系统  

  ---

  #### **1. 项目概述**

  本项目旨在重构平台的用户与权限管理系统，建立一个基于“项目-角色-权限点”的精细化权限控制模型。核心目标是实现权限的灵活分配、流程化申请与审批、权限回收机制，并提供清晰的自助权限查看与管理功能，同时提升管理员的管理效率，保障系统安全与数据隔离。

  ---

  #### **2. 核心目标**

  - 明确定义系统角色、权限点及其边界，确保权限可细粒度控制。
  - 实现权限与具体项目绑定，确保数据隔离与安全。
  - 提供用户自助申请权限、管理员（项目级/系统级）分级审批流程。
  - 提供完整的权限回收机制，支持管理员主动撤销权限。
  - 优化用户界面，提供权限可视化视图及申请状态跟踪。
  - 建立完整的审计日志，记录所有权限变更操作，满足合规与追溯需求。

  ---

  #### **3. 详细需求**

  ##### **3.1 用户与角色定义**

  - **超级管理员 (Admin)**  
    - 内置唯一账户：`admin`。  
    - 拥有系统级超级权限，可创建项目、为任何用户在任意项目上授予或回收任何角色权限，并审批系统级权限申请（如项目创建）。  
    - 不可被删除，可管理其他所有用户。

  - **项目管理员 (Project Admin)**  
    - 每个项目至少有一名项目管理员（通常由项目创建者自动担任，或由超级管理员指定）。  
    - 拥有所负责项目的管理权限，包括审批加入该项目的权限申请、在该项目内为用户授予或回收角色（除项目管理员角色本身外）。  
    - 项目管理员本身可只以是Architect、Builder、Builder、Scribe  角色的兼任者。

  - **普通注册用户**  
    - 初始状态无任何项目权限。  
    - 可申请获得特定项目下的特定角色。  
    - 可查看自己的权限列表及历史申请记录。

  ##### **3.2 权限模型：基于项目的角色与权限点控制**

  权限由“**项目**”、“**角色**”和“**权限点**”三个维度共同决定。**角色**是**权限点**的集合，用户在某个项目中担任某个角色，即拥有该角色所关联的所有权限点。

  **系统预定义权限点示例：**

  | 权限点标识           | 描述             | 所属模块 |
  | -------------------- | ---------------- | -------- |
  | `project:create`     | 创建项目         | 项目     |
  | `project:edit`       | 修改项目信息     | 项目     |
  | `project:delete`     | 删除项目         | 项目     |
  | `story:create`       | 新建用户故事     | 用户故事 |
  | `story:edit`         | 修改用户故事     | 用户故事 |
  | `story:delete`       | 删除用户故事     | 用户故事 |
  | `story:view`         | 查看用户故事     | 用户故事 |
  | `wiki:create`        | 新建Wiki页面     | Wiki     |
  | `wiki:edit`          | 编辑Wiki页面     | Wiki     |
  | `wiki:delete`        | 删除Wiki页面     | Wiki     |
  | `wiki:view`          | 查看Wiki页面     | Wiki     |
  | `issue:create`       | 新建问题单       | 问题单   |
  | `issue:edit`         | 修改问题单       | 问题单   |
  | `issue:delete`       | 删除问题单       | 问题单   |
  | `issue:view`         | 查看问题单       | 问题单   |
  | `project:assignRole` | 在项目内分配角色 | 项目权限 |

  **系统预定义角色及权限点集合：**

  | 角色                           | 权限点集合（项目内）                                         | 备注                                       |
  | ------------------------------ | ------------------------------------------------------------ | ------------------------------------------ |
  | **Architect (架构师)**         | `project:edit`, `project:delete` <br> `story:create`, `story:edit`, `story:delete`, `story:view` <br> `wiki:create`, `wiki:edit`, `wiki:delete`, `wiki:view` <br> `issue:create`, `issue:edit`, `issue:delete`, `issue:view` | 拥有项目几乎所有操作权限（除项目管理权限） |
  | **Validator (验证员)**         | `issue:create`, `issue:edit`, `issue:delete`, `issue:view` <br> `story:edit`, `story:view` <br> `wiki:create`, `wiki:edit`, `wiki:delete`, `wiki:view` | 侧重问题单与Wiki                           |
  | **Builder (构建员)**           | `issue:edit`, `issue:view` <br> `story:edit`, `story:view` <br> `wiki:create`, `wiki:edit`, `wiki:delete`, `wiki:view` | 侧重Wiki，对问题单仅可编辑/查看            |
  | **Scribe (记录员)**            | `wiki:create`, `wiki:edit`, `wiki:delete`, `wiki:view`       | 仅限Wiki操作                               |
  | **Project Admin (项目管理员)** | 包含Architect的所有权限，并额外拥有 `project:assignRole`     | 可在项目内为用户分配角色                   |
  | **Super Admin (超级管理员)**   | 所有权限点（包括跨项目的所有操作）                           | 系统级，不受项目限制                       |

  > **说明**：以上角色与权限点为初始预定义，后续可根据业务需要动态调整权限点与角色的关联。

  ##### **3.3 核心流程**

  **3.3.1 权限申请与审批流程**

  1. **申请入口**：已登录的普通用户通过系统内“权限申请”菜单发起申请。
  2. **填写申请**：
     - 选择目标“项目”（系统自动过滤出用户尚未拥有权限的项目）。
     - 选择希望获得的“角色”（Architect, Builder, Validator, Scribe之一）。
     - 填写“申请理由”（可选，用于辅助审批人判断）。
  3. **提交审批**：申请提交后，生成一条申请记录，状态为“待审批”。
  4. **管理员审批**：
     - **分级审批规则**：
       - 若申请的是非项目管理员角色，且目标项目已有项目管理员，则申请自动流转至该项目管理员的待审批列表。
       - 若目标项目尚无项目管理员，或申请的是项目管理员角色本身，则申请流转至超级管理员`admin`。
     - `admin`或项目管理员在“权限审批”后台查看待处理申请，可批准或拒绝，并可填写审批备注（拒绝时建议必填）。
  5. **生效通知**：审批通过后，该用户立即在对应项目中获得所申请角色的权限，并收到系统通知（站内信/邮件）。若拒绝，用户收到拒绝通知及理由。

  **3.3.2 权限回收流程**

  1. **触发方式**：
     - 管理员主动回收：项目管理员或超级管理员在用户管理/权限管理界面，选择特定用户在项目中的角色，执行“移除角色”操作。
     - 用户离职/账号禁用：触发自动回收该用户在所有项目中的角色。
  2. **确认与执行**：系统二次确认后，立即解除该用户在指定项目中的指定角色关联。
  3. **通知与日志**：权限回收操作记录审计日志，并可选通知被回收权限的用户。

  **3.3.3 项目创建流程**

  1. 任何用户可申请创建项目（需填写项目名称、描述等信息）。
  2. 申请提交后，由超级管理员`admin`审批。
  3. 审批通过后，项目创建成功，**申请人自动成为该项目的第一任项目管理员**。

  ##### **3.4 功能需求列表**

  - **FR1. 权限申请功能**  
    - 提供可视化界面，供用户选择项目（过滤已有权限项目）、选择角色、填写理由并提交申请。  
    - 支持查看自己提交的申请记录及当前状态（待审批/已批准/已拒绝）。  
    - 对于已批准的申请，可查看获得的角色及权限说明。

  - **FR2. 权限审批后台**  
    - 为`admin`和项目管理员提供专属管理界面，区分待处理申请、已处理申请。  
    - 待处理申请列表显示申请人、申请项目、申请角色、申请时间、申请理由。支持批量批准/拒绝。  
    - 审批时可填写备注，拒绝时强制要求填写原因。  
    - 支持按项目、角色、申请人、时间范围筛选和搜索历史审批记录。  
    - 审批人可查看申请人当前已有的权限，辅助决策。

  - **FR3. 权限查看与管理页面**  
    - 用户点击头像下拉菜单中的“我的权限”，进入个人权限视图。  
    - 展示用户在所有项目中拥有的角色，并可将角色展开为具体的权限点列表（如“您在A项目中为架构师，可以：创建项目、编辑项目、管理用户故事…”）。  
    - 提供“申请新权限”快捷入口。  
    - 对于项目管理员，提供“管理项目成员”入口，可进入成员权限管理界面，进行角色授予/回收操作。

  - **FR4. 个人设置**  
    - 用户头像下拉菜单中的“个人设置”用于修改头像、密码、绑定邮箱/手机等个人资料。  

  - **FR5. 权限回收功能**  
    - 项目管理员/超级管理员可在成员管理界面选择用户，移除其在当前项目中的指定角色。  
    - 提供批量移除功能。  
    - 执行前需二次确认，避免误操作。  

  - **FR6. 权限校验中间件**  
    - 所有业务操作（如创建项目、修改问题单、编辑Wiki等）必须经过统一权限校验服务。  
    - 服务端根据当前用户、当前操作所属项目、所需权限点，校验用户是否拥有对应权限。  
    - 校验失败返回403状态码及明确错误信息。  
    - 所有查询接口强制要求传入项目ID，防止跨项目越权访问。

  - **FR7. 审计日志**  
    - 记录所有权限相关的操作，包括：权限申请、审批、授予、回收、角色变更等。  
    - 每条日志至少包含：操作时间、操作人ID、操作人IP、操作类型（grant/revoke/apply/approve/reject）、目标用户ID、目标项目ID、目标角色、审批备注（如有）、操作结果（成功/失败）。  
    - 审计日志需持久化存储，支持按时间、操作人、目标用户等维度检索，供合规审计。

  ---

  #### **4. 非功能性需求**

  - **安全性**  
    - 所有权限检查必须在服务端执行，禁止仅依赖前端控制。  
    - 敏感操作（如权限授予/回收）需进行二次身份验证（如输入密码或短信验证码）?（可选，可后续迭代）。  
    - 防止CSRF、XSS攻击，遵循平台安全规范。  

  - **数据隔离**  
    - 所有数据查询接口必须强制要求项目ID，且服务端校验当前用户在目标项目下是否拥有`view`类权限点。  
    - 确保用户无法通过修改请求参数访问其他项目数据。  

  - **性能**  
    - 权限校验应采用缓存策略（如Redis缓存用户权限集），减少数据库查询。  
    - 权限变更时主动失效相关缓存。  
    - 接口响应时间：权限校验平均耗时 < 50ms（含缓存命中）。  

  - **可用性**  
    - 申请与审批流程简单明了，操作步骤不超过3步。  
    - 审批后台支持批量操作，提升效率。  
    - 提供清晰的错误提示与引导。  

  - **可扩展性**  
    - 权限点与角色支持动态配置，可通过后台管理界面进行调整（未来迭代）。  
    - 角色与权限点的关系可灵活修改，不影响现有业务逻辑。  

  - **审计合规**  
    - 审计日志保留时间不少于1年（可根据法规调整）。  
    - 日志不可篡改，需加密存储并定期备份。
